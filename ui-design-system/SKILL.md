# UI Design System

## ⚠️ 重要：交付前必须验证

**在宣布任何设计任务完成之前，必须执行"阶段 8: 交付验证"**:

- ✅ 对照原始需求文档逐项检查
- ✅ **验证没有添加不在需求中的内容（禁止画蛇添足）**
- ✅ **验证所有创建的导航项/按钮都有完整的内容和功能**
- ✅ **确认不存在空白页面或点击无反应的按钮**
- ✅ **确认需求中定义的所有交互（hover/click/tooltip/modal）都已完整实现**
- ✅ 验证所有导航/按钮能正确工作
- ✅ 确认没有占位内容或 TODO
- ✅ 确认所有功能已完整实现

**不要因为"看起来完成了"就跳过验证** —— 这会导致交付不完整的结果。

**⚠️ 三个关键禁令**:
1. **禁止画蛇添足** - 只实现需求文档中明确要求的内容
2. **禁止空实现** - 创建的任何页面、按钮、导航项都必须有完整的内容和功能
3. **禁止视觉欺诈** - 不允许存在看起来可交互但实际无功能的元素（如只有样式的按钮、用 title 代替自定义 tooltip 等）

## Overview

完整的 UI 设计系统生成流程，从需求分析到可交互设计规范页面。本 skill 不是组件模板库，而是**设计方法论**——教 AI 如何根据产品定位和用户需求做出合适的设计选择。

**核心原则**:
- 根据需求动态设计，不使用固定模板
- 避免总是做纯黑/纯白界面
- 参考竞品和行业标准
- 建立完整的设计系统
- **交付前必须验证完整性**
- **所有输出文档和页面默认使用中文（除非用户明确指定其他语言）**
- **⚠️ 严格禁止画蛇添足：只实现需求文档中明确要求的内容，不添加任何不在需求中的功能、页面或按钮**
- **⚠️ 实施完整性要求：所有创建的页面、按钮、导航项都必须有完整的内容和交互，不允许存在空页面或无功能的按钮**
- **⚠️ 交互功能完整性：需求中定义的所有交互（hover、click、tooltip、modal 等）都必须完整实现，不允许用浏览器默认效果（如 title 属性）代替自定义交互组件**

## 设计思维流程

### 阶段 1: 需求分析

在开始设计之前，必须先理解：

**关键问题**:
1. **产品类型** - Web 应用 / 移动应用 / 桌面软件 / 设计系统？
2. **目标用户** - B2C / B2B / 开发者 / 设计师 / 企业用户？
3. **行业特点** - 金融 / 医疗 / 社交 / 电商 / 工具？
4. **品牌定位** - 专业 / 亲和 / 创新 / 可靠 / 极简？
5. **竞品参考** - 同类产品有哪些设计特点？

**输出**: 需求分析文档

---

### 阶段 2: 风格定位

**⚠️ 避免 AI 陷阱**:
- ❌ 不要总是做纯黑/纯白界面
- ❌ 不要总是用紫色渐变
- ❌ 不要盲目跟风

**正确流程**:

1. **分析行业色彩特征**
   - 金融: 蓝色、绿色（信任、稳定）
   - 社交: 品牌色（友好、活力）
   - 开发者工具: 深色（专业、专注）
   - 电商: 橙色、红色（行动、紧迫）
   - 医疗: 青色、绿色（健康、平静）

2. **确定色彩情绪**
   - 冷色调: 专业、科技、冷静
   - 暖色调: 友好、温暖、活力
   - 中性色: 极简、高级、百搭

3. **建立色彩系统**
   - 主色: 用于核心交互（按钮、链接、激活状态）
   - 辅助色: 用于信息层级
   - 功能色: 成功、警告、错误
   - 中性色: 背景、边框、文字

**输出**: 配色方案 + 设计令牌

**详见**: [color-theory.md](references/color-theory.md)

### 阶段 3: 设计系统定义

#### 字体系统

**选择原则**:
- 中文: 思源黑体、苹方、阿里巴巴普惠体
- 英文: Inter、SF Pro、Roboto（避免滥用）
- 等宽: SF Mono、JetBrains Mono（代码）

**字号层级**:
```
12px - 辅助信息
13px - 次要文字
14px - 正文
15px - 输入框
16px - 小标题
18px - 页面标题
24px+ - 特大标题
```

#### 间距系统

**4px 基础单位**:
```
4px  - 最小间距
8px  - 紧凑间距
12px - 标准间距
16px - 舒适间距
20px - 宽松间距
24px - 大间距
32px+ - 特大间距
```

#### 圆角系统

```
4px  - 气泡指向角
8px  - 小按钮、标签
12px - 卡片
16px - 大按钮、输入框
24px - 丸型按钮
50%  - 圆形元素
```

**输出**: `tokens.md`

### 阶段 4: 组件设计模式

**不是固定组件，而是设计模式**

根据需求设计组件时考虑：

1. **布局组件**
   - Container / Wrapper
   - Header / Sidebar / Footer
   - Grid / Flex 布局

2. **反馈组件**
   - Button / Toggle / Checkbox
   - Input / Select / Textarea
   - Modal / Tooltip / Toast

3. **展示组件**
   - Card / List / Table
   - Avatar / Badge / Tag
   - Chart / Image / Video

4. **导航组件**
   - Menu / Breadcrumb / Tabs
   - Pagination / Sidebar

**详见**: [component-patterns.md](references/component-patterns.md)

**输出**: `components.md`

### 阶段 5: 交互与动画

#### 动画时长

```
0.1s - 微交互（悬停、点击）
0.2s - 快速过渡
0.3s - 标准动画
0.4s - 中等动画
0.5s+ - 缓慢动画（谨慎使用）
```

#### 缓动函数

```
ease-out - 退出动画（推荐）
ease-in-out - 往返动画
ease-in - 进入动画（少用）
linear - 循环动画
cubic-bezier - 弹性效果
```

#### 交互状态

每个可交互元素需要定义：
- Default（默认）
- Hover（悬停）
- Active/Focus（激活）
- Disabled（禁用）
- Loading（加载中）
- Error（错误）

**输出**: `interactions.md`

### 阶段 8: 交付验证 ⚠️ **关键步骤**

**在宣布任务完成之前，必须执行以下验证**:

#### ⚠️ 禁止画蛇添足检查

**【关键】** 在开始验证之前，先检查是否添加了不在需求中的内容：

- [ ] 检查所有导航项/菜单是否都在需求文档中明确列出
- [ ] 检查所有页面是否都在需求文档中明确要求
- [ ] 检查所有按钮/功能是否都在需求文档中明确描述
- [ ] 如发现额外内容，要么删除，要么补充完整实现并更新需求确认

**原则：只实现需求文档中明确要求的内容，不添加任何未经确认的功能。**

#### 验证步骤

1. **需求对照检查**
   - 重新阅读原始需求文档
   - 列出所有需求点（模块、页面、功能、交互）
   - 逐一检查每个需求是否已实现
   - **特别检查：需求中明确定义的交互行为（hover、click、tooltip、modal 等）**

2. **实施完整性检查**
   - [ ] 检查所有导航项是否都有对应的页面/内容
   - [ ] 检查所有按钮是否都有正确的功能/交互
   - [ ] 检查所有页面是否都有完整的内容（不是空白）
   - [ ] 确认没有任何"点击无反应"的交互元素
   - [ ] 确认没有任何"空白页面"

3. **⚠️ 交互功能完整性检查（关键新增）**

   **这是最常被忽视的检查项 - 必须逐个验证每个交互元素**

   - [ ] **需求中定义的所有 Hover 交互都已实现**
     - 例如：KPI info 图标 hover 显示详细说明
     - 不能只用浏览器默认的 title 属性代替自定义 tooltip

   - [ ] **需求中定义的所有 Click 交互都已实现**
     - 例如：按钮点击跳转、显示弹窗、执行操作等
     - 不存在"看起来可点击但点击无反应"的按钮

   - [ ] **所有 Tooltip 组件都已完整实现**
     - 有完整的样式、定位、内容
     - 不是简单的 title 属性

   - [ ] **所有 Modal/弹窗组件都已完整实现**
     - 有打开、关闭、确认、取消等完整功能
     - 不是只有视觉外壳

   - [ ] **所有表格操作按钮都有对应功能**
     - 编辑、查看、删除等按钮都能正确响应
     - 不是只占位无功能

   **禁止"视觉欺诈"**：
   - ❌ 不允许存在看起来可交互但实际无功能的元素
   - ❌ 不允许用浏览器默认效果（如 title）代替自定义交互组件
   - ❌ 不允许只做按钮样式不做按钮功能

4. **interactive.html 验证**
   - [ ] 所有导航项可点击且切换到正确内容
   - [ ] 所有场景/页面都能正确显示
   - [ ] 所有交互元素有正确的行为
   - [ ] 没有重复或占位内容
   - [ ] 所有动画流畅运行
   - [ ] 没有点击后显示空白内容的导航项
   - [ ] 没有点击后无反应的按钮
   - [ ] **所有需求中定义的交互都已实现（hover/click/tooltip/modal）**

5. **design-guide.html 验证**
   - [ ] 左侧导航的所有项都能点击
   - [ ] 每个组件都有完整的预览
   - [ ] 每个组件都有规范的说明
   - [ ] 所有组件数据完整（不是示例/占位）
   - [ ] 没有缺失的组件

5. **文档完整性验证**
   - [ ] tokens.md 包含所有设计令牌
   - [ ] components.md 包含所有组件文档
   - [ ] interactions.md 包含所有动画规范
   - [ ] design-spec.md 完整
   - [ ] dev-guide.md 完整
   - [ ] README.md 完整

#### 交叉验证清单

| 需求中的模块 | interactive.html | design-guide.html | 状态 |
|------------|------------------|-------------------|-----|
| 模块1 | ✅/❌ | ✅/❌ | ... |
| 模块2 | ✅/❌ | ✅/❌ | ... |
| ... | ... | ... | ... |

#### 常见陷阱检查

- [ ] 没有使用"TODO"、"示例"、"待填充"等占位内容
- [ ] 没有只实现部分功能就声称完成
- [ ] 没有因为"时间不够"而简化实现
- [ ] 所有承诺的交付物都已创建
- [ ] 没有添加不在需求中的导航项/页面/按钮
- [ ] 没有创建后没有实现内容的空白页面
- [ ] 没有点击后无反应的按钮或链接
- [ ] **【新增】没有"视觉欺诈"元素（看起来可交互但实际无功能）**
- [ ] **【新增】没有用浏览器默认效果（如 title 属性）代替自定义交互组件**
- [ ] **【新增】需求中定义的所有交互（hover/click/tooltip/modal）都已完整实现**

**验证通过标准**:
- 所有需求点都已实现
- 所有交互功能正常工作
- 所有组件数据完整
- 没有占位或不完整的内容
- 所有导航项都有对应的页面内容
- 所有按钮都有正确的功能实现
- 没有任何空白页面或无功能的交互元素
- 只包含需求文档中明确要求的内容（无额外添加）
- **【新增】需求中定义的每个交互都已完整实现**
- **【新增】不存在"视觉欺诈"元素**

**如果验证失败**:
1. 记录缺失的内容
2. 补充完整实现
3. 重新验证
4. 只有验证通过才能宣布完成

## 完整工作流

### 单页面应用设计

```
1. 需求分析
   ├─ 产品类型、目标用户、行业特点
   └─ 参考竞品（2-3 个）

2. 风格定位
   ├─ 确定配色方案
   ├─ 选择字体系统
   └─ 建立设计令牌

3. 布局设计
   ├─ 页面结构（Header / Sidebar / Content）
   ├─ 响应式断点
   └─ 网格系统

4. 组件设计
   ├─ 按需设计组件（不是使用模板）
   ├─ 定义组件状态
   └─ 编写组件代码

5. 交互设计
   ├─ 动画效果
   ├─ 交互状态
   └─ 手势操作

6. 原型开发
   ├─ 创建 interactive.html（⚠️ 只实现需求中明确的页面）
   ├─ 实现场景演示
   └─ 测试交互流程

7. 文档输出
   ├─ 创建 5 个核心文档
   ├─ 创建 design-guide.html
   └─ 创建 README.md

8. **交付验证** ⚠️ 必须执行
   ├─ 检查是否添加了不在需求中的内容
   ├─ 对照需求文档逐项检查
   ├─ 验证所有导航项/按钮都有对应内容
   ├─ 验证 interactive.html 所有功能
   ├─ 验证 design-guide.html 所有组件
   ├─ 确保没有占位或不完整内容
   ├─ 确保没有空白页面或无功能按钮
   └─ 验证通过后才能宣布完成
```

### 设计系统设计

```
1. 理解产品/品牌
   ├─ 品牌调性
   ├─ 设计原则
   └─ 目标受众

2. 建立设计令牌
   ├─ 颜色系统
   ├─ 字体系统
   ├─ 间距系统
   └─ 组件库定义

3. 设计核心组件
   ├─ 按优先级排序
   ├─ 定义组件 API
   └─ 创建示例

4. 编写设计规范
   ├─ 完整的组件文档
   ├─ 使用指南
   └─ 最佳实践

5. 创建设计资源
   ├─ Sketch/Figma 库（如需要）
   ├─ 代码模板
   └─ 示例项目

6. **交付验证** ⚠️ 必须执行
   ├─ 对照设计需求逐项检查
   ├─ 验证所有组件已实现
   ├─ 确保文档完整无误
   └─ 验证通过后才能宣布完成
```
